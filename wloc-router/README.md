# wloc-router

Small router-side Rust MITM service for Apple WLOC responses. It is intentionally narrow: it only accepts `gs-loc.apple.com` and `gs-loc-cn.apple.com`, and only modifies `POST /clls/wloc` responses.

## What it does

- Terminates TLS locally with a user-installed CA.
- Proxies Apple WLOC requests to the original Apple host.
- Scans Apple WLOC protobuf wire data and rewrites latitude, longitude, and accuracy.
- Supports `/wloc-settings/save` compatible query parameters:
  - `?lon=...&lat=...&acc=...`
  - `?action=query`
  - `?action=clear`

## Build

```sh
cargo build --release
```

For OpenWrt x86_64, build on a Linux host with a suitable target/toolchain, then copy the binary to `/usr/bin/wloc-router`.

## GitHub Actions build

This repository includes `.github/workflows/wloc-router.yml`. After pushing to GitHub, the workflow will:

- run `cargo test` and `cargo clippy`;
- build release binaries for `x86_64-unknown-linux-gnu` and `x86_64-unknown-linux-musl`;
- upload `.tar.gz` artifacts containing the binary, deploy files, and this README.

Use the Actions tab or the manual `workflow_dispatch` trigger when you want a fresh router binary without installing Rust locally.

Example cross-build from Linux for a glibc x86_64 OpenWrt image:

```sh
rustup target add x86_64-unknown-linux-gnu
cargo build --release --target x86_64-unknown-linux-gnu
```

For musl-based images, use `x86_64-unknown-linux-musl` and verify the resulting binary runs on the router before enabling the init script.

## Install on OpenWrt

```sh
mkdir -p /etc/wloc-router
cp deploy/config.toml /etc/wloc-router/config.toml
cp target/release/wloc-router /usr/bin/wloc-router
chmod +x /usr/bin/wloc-router

wloc-router gen-certs --out-dir /etc/wloc-router
cp deploy/wloc-router.init /etc/init.d/wloc-router
chmod +x /etc/init.d/wloc-router
/etc/init.d/wloc-router enable
/etc/init.d/wloc-router start
```

Install and fully trust `/etc/wloc-router/certs/ca.pem` on the Apple device.

## Routing

Only route selected client devices to this service. For example, for client `192.168.1.37`, DNS or transparent redirect should send these names to OpenWrt:

```text
gs-loc.apple.com
gs-loc-cn.apple.com
```

The service listens on `192.168.1.254:9443` by default. If DNS is hijacked globally for these names, configure `upstream_resolve` with real Apple edge IPs to prevent the service from connecting back to itself.

OpenWrt firewall/redirect rules are intentionally not generated by this project. Keep routing scoped to selected client IPs, for example `192.168.1.37`, so other Apple devices continue using normal Apple location services.

See `deploy/openwrt-routing.md` for source-scoped routing notes and loop-prevention options.

## Verify

```sh
curl -k --resolve gs-loc.apple.com:9443:192.168.1.254 \
  "https://gs-loc.apple.com:9443/wloc-settings/save?action=query"
```

Use logs to confirm `patched wloc response` appears after the Apple device sends `/clls/wloc`.
